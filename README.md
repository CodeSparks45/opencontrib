## 💀 The Problem — Why OpenContrib Exists

<br/>

> If you've ever participated in **GSSoC**, **Hacktoberfest**, **JWOC**, **SSOC**, or any open-source program, you have felt this pain personally. This section is for you.

<br/>

### 🔴 Problem 1 — The 80/20 Time Trap

**Kya (What):** Developers spend **80% of their time searching** for a suitable issue and only **20% actually writing code.**

**Kyu (Why):** GitHub's native search is broken for contributors. You filter by `good first issue`, get 400 results, and have zero context about which one is actually available, beginner-friendly, or worth your time.

**Kab (When):** Every. Single. Time. you open GitHub with the intention to contribute.

**Kaise (How it hurts):** You lose momentum before writing a single line. The dopamine of contribution never arrives. You close the tab and open YouTube instead.

**Solution ✅:** OpenContrib's AI-powered feed does the searching for you. Open the app → see perfectly matched issues immediately. **Zero wasted search time.**

---

### 🔴 Problem 2 — The "Already Assigned" Heartbreak

**Kya (What):** You find the perfect issue. You read through the codebase for 20 minutes. You write a thoughtful comment: *"Hey, can I work on this?"* — and get back: *"Sorry, this has already been assigned."*

**Kyu (Why):** Popular repos under GSSoC/Hacktoberfest get flooded. A single good issue gets 20–30 comments in the first hour. By the time you find it through manual search, it's long gone.

**Kab (When):** The worst moment — when you're **pumped up and ready to code** but get immediately deflated by a crowded thread.

**Kaise (How it hurts):** Kills motivation. Makes open source feel exclusive and unfair. Especially brutal for beginners who are trying to build their GitHub profile.

**Solution ✅:** OpenContrib's **Sniper Mode** strictly fetches only `comments:0` + `no:assignee` issues via the GitHub API. By definition, **no one has touched them yet.** You're always first.

---

### 🔴 Problem 3 — No Skill Calibration

**Kya (What):** There's no reliable way to know if an issue is actually at your level. A "good first issue" tag on `linux/kernel` hits very differently than one on a beginner JavaScript project.

**Kyu (Why):** Labels are applied manually by maintainers who have wildly different standards. The tag system is completely subjective and unreliable.

**Kab (When):** When you click into an issue expecting something doable and are immediately hit with 800 lines of C++ template metaprogramming.

**Kaise (How it hurts):** Imposter syndrome. Beginners bounce off issues that are labelled beginner but require senior-level knowledge. They assume they're not good enough. They stop trying.

**Solution ✅:** Our **AI Difficulty Radar** analyzes the codebase complexity, required domain knowledge, and issue description to auto-score every issue from `Beginner → Intermediate → Advanced`. No more guessing.

---

### 🔴 Problem 4 — Zero Developer Momentum

**Kya (What):** Open source contribution is a habit, not a one-time event. But there's nothing keeping you consistent.

**Kyu (Why):** No accountability. No rewards. No progress tracking. You contribute once, forget about it, and come back 3 months later to find your GitHub heatmap is a desert.

**Kab (When):** Every time life gets busy and "I'll contribute tomorrow" becomes never.

**Kaise (How it hurts):** Your GitHub profile stays empty. Job recruiters scroll past. You miss the network effect of being a visible contributor in popular projects.

**Solution ✅:** OpenContrib's **Streak System + XP + Leaderboard** turns daily contribution into a game. Miss a day, break your streak. Hit a milestone, earn XP. Climb the leaderboard. The psychological hooks that make Duolingo addictive — applied to open source.

---

### 🔴 Problem 5 — PR Chaos Across Repos

**Kya (What):** Once you start contributing to multiple repos, tracking your open PRs becomes a nightmare. PR under review in `facebook/react`. Another open in `vercel/next.js`. Another merged somewhere you forgot.

**Kyu (Why):** GitHub's native PR dashboard shows PRs you **authored** but there's no clean, unified view across organizations when you're a contributor (not a maintainer).

**Kab (When):** When a maintainer asks for changes and you don't notice for a week because it was buried in GitHub notifications.

**Kaise (How it hurts):** PRs go stale. Maintainers close them. Your contribution never gets counted. Wasted effort.

**Solution ✅:** OpenContrib's **Unified PR Tracker** pulls all your open/review/merged PRs across every repository into one clean dashboard with status indicators. Never miss a review request again.

## 💡 The Solution — Sniper Mode Discovery

<br/>

OpenContrib is a **precision-engineered, AI-powered contribution platform** that kills every problem above. Here's the core philosophy:

```
Traditional GitHub Search:
  You → Search GitHub → Filter → Scroll 400 results → Find issue → Already assigned → Repeat ♻️

OpenContrib:
  You → Open App → AI fetches 0-comment, unassigned issues → Matched to YOUR stack → Swipe → Contribute ⚡
```

The engine works in real-time. No stale data. No cached garbage. Every issue shown is:

- ✅ **`comments: 0`** — You're literally the first person looking at it
- ✅ **`no:assignee`** — No one has claimed it
- ✅ **AI-matched** — Scored against your GitHub profile and language history
- ✅ **Difficulty-scored** — Beginner / Intermediate / Advanced, actually calibrated
- ✅ **Scope-switchable** — Toggle between GSSoC-specific or worldwide open source

## ✨ Features — God-Level Breakdown

<br/>

### 🔫 Feature 1 — Real-Time Sniper Mode

The core. The heartbeat. The thing that makes everything else possible.

```
GitHub REST API v3
  └── GET /search/issues
        ├── q: is:open is:issue comments:0 no:assignee label:"good first issue"
        ├── sort: created (newest first)
        ├── per_page: 30
        └── [Optional] org:GirlScriptSummerOfCode → GSSoC Mode
```

- Fetches live, fresh issues every time you load the feed
- **GSSoC Toggle:** Switch between contributing to GSSoC-tagged repositories vs the entire open-source ecosystem
- No database caching — data is always real-time from GitHub's API
- Smart rate-limit handling with retry logic and user-facing feedback

---

### 🧠 Feature 2 — AI Issue Matching (Powered by Claude)

Most "recommendation systems" are just keyword matching dressed up with a fancy name. Ours isn't.

**How it actually works:**

```
Step 1: Read your GitHub profile
  └── Languages used, repos contributed to, PRs merged, issue comments

Step 2: Build a skill vector
  └── TypeScript: Expert | React: Advanced | Rust: Beginner | Go: Never touched

Step 3: Score every fetched issue against your vector
  └── Issue in Rust codebase → Beginner match for you → Show with 60% score
  └── Issue in TypeScript codebase → Expert match → Show with 94% score

Step 4: Re-rank the feed
  └── Highest match scores bubble to the top
```

- Personalized from day one using your existing GitHub history
- No manual setup, no "tell us your languages" forms
- Continuously improves as you save, skip, and contribute

---

### ⚡ Feature 3 — Swipe Mode (Tinder for Devs)

High-speed, addictive issue discovery that respects your time.

| Action | What happens |
|--------|-------------|
| 👉 Swipe Right / Press `V` | Issue saved to your personal Vault |
| 👈 Swipe Left / Press `X` | Issue dismissed, never shown again |
| ⭐ Press Star | Issue bookmarked for later with notification |
| `Space` | Quick-preview expanded view without leaving feed |

- Mobile-first touch gestures with desktop keyboard shortcuts
- Smooth spring animations powered by CSS cubic-bezier transitions
- Your vault persists across sessions (localStorage + optional cloud sync)

---

### 🎮 Feature 4 — Gamified Developer Workflow

The psychological infrastructure that keeps you coming back.

#### XP System
```
Save an issue          → +10 XP
Open issue in GitHub   → +25 XP
Open a PR              → +100 XP
PR gets merged         → +500 XP  🎉
Maintain a 7-day streak → +250 XP bonus
Hit a new Level        → Badge unlocked
```

#### Streak System
- Daily goal: interact with at least 1 issue
- Miss a day → streak resets to 0 (just like Duolingo — it hurts, intentionally)
- Streak shields earned via XP can protect one missed day per month

#### Global Leaderboard
- Ranked by **verified merged PRs** (not just XP — no gaming the system)
- Weekly + All-time rankings
- GitHub username + avatar displayed publicly (opt-out available)
- Filter by tech stack: "Top React Contributors This Week"

---

### 💎 Feature 5 — Ultra-Premium UI/UX

Because developers deserve beautiful tools.

- **Dark-mode first** — because obviously
- **Bento Grid layout** — information-dense without feeling cluttered
- **Canvas particle system** — animated network effect background (60fps, GPU-accelerated)
- **Glassmorphism cards** — `backdrop-filter: blur()` with layered gradients
- **Micro-interactions everywhere** — hover tilts, magnetic buttons, spring animations
- **Bricolage Grotesque font** — distinctive, premium, not the same boring Inter everyone uses
- **Live activity feed** — real-time pulse of what contributors are doing globally
- **Responsive** — works perfectly from 320px mobile to 4K ultrawide

---

### 🗺️ Feature 6 — Unified PR Dashboard

One dashboard. Every PR. Every repo.

```
Status Indicators:
  🟢 Open      — Awaiting maintainer review
  🟡 In Review — Maintainer has responded, changes requested
  🟣 Merged    — Successfully landed. +XP credited.
  🔴 Closed    — Did not merge. Learn & move on.
```

- Pulls from GitHub API using authenticated user token
- Shows PR title, repo, time since last activity, and review status
- One-click jump to the PR on GitHub
- Notifies you (in-app) when a review is requested

| **Deployment** | [Vercel](https://vercel.com/) | Zero-config Next.js deploys. Edge functions for API routes. Free tier is generous. |
| **Language** | TypeScript (strict) | No runtime surprises. Full type safety across API responses and components. |

## 🚀 Quick Start — Run Locally in 5 Minutes

<br/>

### Prerequisites

Make sure you have these installed before anything else:

```bash
node --version    # v18.17.0 or higher required
npm --version     # v9.0.0 or higher
git --version     # Any recent version
```

Don't have Node.js? Download it from [nodejs.org](https://nodejs.org/) (LTS version recommended).

---

### Step 1 — Clone the Repository

```bash
git clone https://github.com/CodeSparks45/opencontrib.git
cd opencontrib
```

---

### Step 2 — Install Dependencies

```bash
npm install
# OR if you prefer yarn:
yarn install
# OR pnpm (fastest):
pnpm install
```

---

### Step 3 — Create GitHub OAuth App

You need a GitHub OAuth App to enable login. Here's exactly how:

1. Go to [github.com/settings/developers](https://github.com/settings/developers)
2. Click **"New OAuth App"**
3. Fill in the form:

```
Application name:     OpenContrib (Local)
Homepage URL:         http://localhost:3000
Authorization callback URL:  http://localhost:3000/api/auth/callback/github
```

4. Click **"Register application"**
5. Copy your **Client ID**
6. Click **"Generate a new client secret"** and copy the **Client Secret**

> ⚠️ **Important:** Never commit your client secret to git. It goes in `.env.local` only.

---

### Step 4 — Set Up Environment Variables

Create a `.env.local` file in the project root:

```bash
touch .env.local
```

Then open it and add:

```env
# ─── NextAuth.js ──────────────────────────────────────────────────────────────
NEXTAUTH_URL=http://localhost:3000
NEXTAUTH_SECRET=your_super_secret_key_here_make_it_long_and_random

# ─── GitHub OAuth ─────────────────────────────────────────────────────────────
GITHUB_CLIENT_ID=your_github_client_id_here
GITHUB_CLIENT_SECRET=your_github_client_secret_here

# ─── GitHub API Token (for higher rate limits) ────────────────────────────────
# Go to: github.com/settings/tokens → Generate new token (classic)
# Scopes needed: public_repo, read:user
GITHUB_API_TOKEN=ghp_your_personal_access_token_here

# ─── Anthropic Claude API (for AI Issue Matching) ─────────────────────────────
# Get your key at: console.anthropic.com
ANTHROPIC_API_KEY=sk-ant-your_key_here
```

> 💡 **Generate `NEXTAUTH_SECRET`:** Run this in your terminal:
> ```bash
> openssl rand -base64 32
> ```

---

### Step 5 — Run the Development Server

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000) in your browser.

You should see the landing page. Sign in with GitHub and you're live. 🎉

---

### Step 6 (Optional) — Build for Production

```bash
npm run build
npm run start
```

## ⚙️ Environment Variables

<br/>

Complete reference for every environment variable OpenContrib uses:

| Variable | Required | Description | Where to get it |
|----------|----------|-------------|-----------------|
| `NEXTAUTH_URL` | ✅ Yes | Your app's base URL | `http://localhost:3000` locally |
| `NEXTAUTH_SECRET` | ✅ Yes | Random secret for session encryption | `openssl rand -base64 32` |
| `GITHUB_CLIENT_ID` | ✅ Yes | GitHub OAuth App Client ID | [github.com/settings/developers](https://github.com/settings/developers) |
| `GITHUB_CLIENT_SECRET` | ✅ Yes | GitHub OAuth App Client Secret | Same as above |
| `GITHUB_API_TOKEN` | ✅ Yes | Personal Access Token for API calls | [github.com/settings/tokens](https://github.com/settings/tokens) |
| `ANTHROPIC_API_KEY` | ⚡ For AI | Claude API key for issue matching | [console.anthropic.com](https://console.anthropic.com) |

> **Note on rate limits:** Without `GITHUB_API_TOKEN`, you get 60 requests/hour unauthenticated. With a PAT, you get 5,000 requests/hour. Always use a token.

## 🤝 Contributing Guide

<br/>

> We built this for the developer community. Contributions are not just welcome — they're the whole point.

<br/>

### Getting Started

1. **Fork** the repository (top-right button on GitHub)

2. **Clone** your fork:
```bash
git clone https://github.com/YOUR_USERNAME/opencontrib.git
cd opencontrib
```

3. **Create a feature branch:**
```bash
# Use a descriptive name
git checkout -b feat/add-language-filter
git checkout -b fix/streak-reset-bug
git checkout -b docs/improve-contributing-guide
```

4. **Follow the Quick Start guide** above to get it running locally.

---

### Branch Naming Convention

```
feat/       → New feature
fix/        → Bug fix
docs/       → Documentation only
style/      → CSS/UI changes (no logic changes)
refactor/   → Code restructure (no feature change)
perf/       → Performance improvement
test/       → Adding or fixing tests
chore/      → Build process, dependencies
```

---

### Commit Message Convention

We follow [Conventional Commits](https://www.conventionalcommits.org/):

```bash
# ✅ Good commits
git commit -m "feat: add language filter to sniper mode"
git commit -m "fix: streak not resetting at midnight UTC"
git commit -m "perf: reduce GitHub API calls by 40% with SWR caching"
git commit -m "docs: add environment variable reference table"

# ❌ Bad commits
git commit -m "fixed stuff"
git commit -m "changes"
git commit -m "update"
```

---

### Pull Request Checklist

Before opening a PR, make sure:

- [ ] Your code runs without errors (`npm run dev`)
- [ ] Build passes (`npm run build`)
- [ ] TypeScript has no errors (`npx tsc --noEmit`)
- [ ] No new ESLint warnings (`npm run lint`)
- [ ] You've tested on mobile viewport (DevTools responsive mode)
- [ ] Your PR title follows Conventional Commits format
- [ ] You've described **what** changed and **why** in the PR description

---

### Good First Issues 🟢

New to the project? Look for issues tagged:
- [`good first issue`](https://github.com/CodeSparks45/opencontrib/issues?q=label%3A%22good+first+issue%22) — Perfect for first-timers
